# Feishu Docx Engine 0.1.1 / Feishu Markdown Sync 0.6.2 Release Checklist

## Release identity

- Engine: `feishu-docx-engine@0.1.1`.
- CLI: `feishu-md-sync@0.6.2`.
- CLI dependency: exactly `feishu-docx-engine: 0.1.1`.
- Git tag: `v0.6.2`, created only from the merged Release PR commit.
- Release manifest: `.github/releases/v0.6.2.json`.
- Engine publication: enabled and provenance-bound to the same `v0.6.2` tag and commit.

## Included changes

- Code creation retries eventual readback without repeating a successful provider mutation.
- Image-to-Whiteboard replacement accepts a later verified revision while keeping structural, sibling, raw-resource, and unrelated-drift checks exact.
- Tracked Supademos preserve existing ISV blocks only when receipt, baseline, local placement, `isShowcase`, semantic locators, and remote identity agree.
- npm provenance verification tolerates bounded registry-attestation propagation and still verifies exact bytes, tag, commit, workflow identity, and Sigstore evidence.

## Upgrade compatibility gate

- Engine 0.1.1 emits `engineVersion: 0.1.1` for newly prepared batches.
- Engine 0.1.1 accepts valid schema-v1 prepared batches and recovery checkpoints emitted by engine 0.1.0.
- Compatibility does not bypass the existing batch fingerprint, compiler-structure, document ID, revision, snapshot hash, preflight, resource evidence, or unrelated-drift checks.
- Future engine versions and unsupported schema versions remain rejected before transport reads or writes as applicable.
- Tests must cover both apply and read-only recovery of 0.1.0 state under 0.1.1, plus rejection of future versions.

## Artifact manifest

1. Build and pack both workspaces twice from the final release tree.
2. Confirm each package is byte-identical across the two packs.
3. Record each tarball's npm SHA-512 integrity and local SHA-256 in `.github/releases/v0.6.2.json`.
4. Confirm the CLI tarball depends on exactly `feishu-docx-engine@0.1.1`.
5. Confirm the CLI consumer smoke installs the engine from the npm registry rather than a local engine tarball.

## Tag and publication gate

After the Release PR is merged and only after explicit approval:

```bash
git tag -a v0.6.2 <approved-merged-release-commit-sha> -m 'feishu-md-sync v0.6.2'
git push origin refs/tags/v0.6.2
```

The protected workflow must:

1. pack bytes identical to the committed manifest;
2. publish and verify `feishu-docx-engine@0.1.1` first;
3. smoke the CLI candidate with the registry-resolved engine;
4. publish and verify `feishu-md-sync@0.6.2`;
5. validate the tagged Agent Skill against the released CLI;
6. create the GitHub Release from `.github/releases/v0.6.2.md`.

Do not publish either candidate manually, move an existing tag, or reuse a package version with different bytes.

## Post-release verification

Run from a clean checkout detached at the immutable `v0.6.2` tag after registry propagation:

```bash
set -euo pipefail
REPO_ROOT="$(git rev-parse --show-toplevel)"
cd "$REPO_ROOT"
test -z "$(git status --porcelain)"
test "$(git rev-parse --abbrev-ref HEAD)" = 'HEAD'
test "$(git describe --tags --exact-match)" = 'v0.6.2'
TEMP_HOME="$(mktemp -d)"
CLI_PREFIX="$(mktemp -d)"
npm view feishu-docx-engine@0.1.1 version dist.integrity dist.shasum dist-tags --json
npm view feishu-md-sync@0.6.2 version dependencies dist.integrity dist.shasum dist-tags --json
npm install --prefix "$CLI_PREFIX" --ignore-scripts --no-audit --no-fund feishu-md-sync@0.6.2
FMS="$CLI_PREFIX/node_modules/.bin/feishu-md-sync"
"$FMS" --version
"$FMS" publish --help
HOME="$TEMP_HOME" XDG_STATE_HOME="$TEMP_HOME/.local/state" \
  npx --yes skills@1.5.17 add 'liyun95/feishu-md-sync#v0.6.2' \
  --skill feishu-md-sync --agent codex --global --copy --yes
node "$REPO_ROOT/scripts/hash-skill-tree.mjs" "$REPO_ROOT/skills/feishu-md-sync"
node "$REPO_ROOT/scripts/hash-skill-tree.mjs" "$TEMP_HOME/.agents/skills/feishu-md-sync"
FEISHU_MD_SYNC_SKILL_DIR="$TEMP_HOME/.agents/skills/feishu-md-sync" \
FEISHU_MD_SYNC_BIN="$FMS" node "$REPO_ROOT/scripts/validate-agent-skill.mjs"
```

Verify both Sigstore provenance bundles against tag `v0.6.2`, the immutable release commit SHA, repository, workflow path, GitHub Actions builder, and push event. Verify the GitHub Release notes match `.github/releases/v0.6.2.md`.

Only after registry bytes, provenance, tagged Skill, GitHub Release, controlled read-only parity, and any separately approved disposable live test pass may Project B consume `feishu-docx-engine@0.1.1`.
